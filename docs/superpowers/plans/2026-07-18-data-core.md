# Data Core Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build a UI-independent TypeScript data core that imports the provisional Clay CSV, preserves provenance, identifies founder candidates safely, and defines the first Supabase schema.

**Architecture:** A standalone `packages/data-core` package owns pure normalization, parsing, deduplication, and founder-candidate resolution. It has no UI or database client dependency, so the Next.js application can consume it later. A root Supabase migration defines persistence, while public Clay pilot output is stored as a labeled, reviewable snapshot rather than silently merged into verified founder data.

**Tech Stack:** TypeScript 5, Node.js 22+, Vitest, `csv-parse`, PostgreSQL/Supabase SQL

## Global Constraints

- Do not modify or scaffold UI files.
- Keep route/controller concerns outside this package; business rules live in pure functions.
- Treat Clay CSV values and MCP results as unverified source data.
- Missing evidence must never be converted into negative evidence.
- A LinkedIn result is not a verified founder relationship until current-company identity is resolved.
- LinkedIn must not be scraped by the application.
- Use `csv-parse` because the supplied descriptions contain quoted commas and embedded newlines; do not implement a custom CSV parser.
- Keep files below 300 lines when reasonable.
- The package must work without Supabase, OpenAI, Clay, or GitHub credentials.
